Covers scenarios that unit + integration tests can't exercise: real Obsidian UI, real vault file I/O, real daemon/plugin WebSocket handshake, real modal interactions. Run each scenario against a built daemon + reloaded plugin.

## Core scenarios

1. **Fresh meeting with Case A first-pass auto-relabel**
   - [ ] Start a 1-on-1 meeting (Teams native) where `metadata.participants` has exactly 1 eligible participant (e.g., `Sean Mooney`).
   - [ ] Record for a few seconds. Stop.
   - [ ] Pipeline runs diarize -> first-pass relabel writes `.speakers.json` -> analyze sees named transcript.
   - [ ] Verify the generated note frontmatter `participants: [[Sean Mooney]]` and transcript display labels in the body are named (not `SPEAKER_00`).

2. **Fresh meeting, Case A doesn't hold (multiple speakers)**
   - [ ] Start a meeting with 2+ distinct speakers where `metadata.participants` has 2+ entries.
   - [ ] Record, stop, pipeline runs.
   - [ ] Transcript display stays `SPEAKER_00`, `SPEAKER_01`, etc. No `.speakers.json` written.

3. **Correct unresolved `SPEAKER_00` to existing People note**
   - [ ] Open a meeting note where `participants` contains `SPEAKER_00` (or transcript still has raw SPEAKER_XX).
   - [ ] Run the "Fix speakers" command. Modal opens; speaker list fetched from daemon (not regex from body).
   - [ ] Modal autocomplete shows existing People notes + known_contacts + aliases.
   - [ ] Type an existing name. Inline hint reads "Links to <canonical>".
   - [ ] Click Save. Modal shows "Speaker corrections submitted -- reprocessing..." and closes.
   - [ ] Reprocess completes. Note frontmatter `participants` updated. Transcript display labels in body are named.
   - [ ] Clip button (daemon side: `/api/recordings/<stem>/clip?speaker_id=SPEAKER_00`) still plays the correct audio.

4. **Rename an already-named speaker (re-correction)**
   - [ ] Open a previously-corrected note where `SPEAKER_00` was mapped to "Alice".
   - [ ] Run Fix speakers. Modal shows "Alice" with the display pre-populated.
   - [ ] Rename to "Bob". Inline hint: "Links to Bob" (if Bob exists) OR "Will create new contact and People note".
   - [ ] Save. Reprocess completes. Frontmatter now shows Bob (and Alice no longer, if replaced). Clip still plays original audio keyed by `speaker_id`.

5. **Create a new contact with People stub**
   - [ ] Type a name not in People notes or known_contacts. Inline hint: "Will create new contact and People note".
   - [ ] Save. Daemon creates both: new entry in `config.yaml known-contacts` (with ruamel-preserved comments + custom blocks) AND `Clients/<org>/People/<name>.md` stub using the canonical `_generate_person_stub` template.
   - [ ] Open the new People note in Obsidian -- verify sections exist ("Key Topics", "Meeting History", etc.).

6. **Near-match accepts suggestion -> adds alias**
   - [ ] Type "Sean M." where `Sean Mooney` is an existing contact. Modal shows inline: "Did you mean Sean Mooney? [Use existing] [Create new anyway]".
   - [ ] Click "Use existing". Plan becomes `link_to_existing`, hint updates.
   - [ ] Save. Verify `known-contacts` in `config.yaml`: Sean Mooney's `aliases` list now contains `Sean M.`.
   - [ ] In a subsequent meeting where enrichment sees "Sean M.", `match_known_contacts` canonicalizes to "Sean Mooney" via alias.

7. **Near-match decline -> create new**
   - [ ] Type "Sean M." again (or a different near-match case). Modal shows suggestion.
   - [ ] Click "Create new anyway". Plan becomes `create_new_contact` or `ineligible` depending on other guards.
   - [ ] If create, save creates new contact. If ineligible (e.g., typed name matches a Company note), hint reads "Rename required: ..." and Save is disabled.

8. **Ineligible: SPEAKER_XX typed**
   - [ ] Type `SPEAKER_00` as a correction target. Hint: "Rename required: SPEAKER_XX".
   - [ ] Save button disabled.

9. **Ineligible: company collision**
   - [ ] Type `Disbursecloud` (a Company note basename). Hint: "Rename required: matches Company note".
   - [ ] Save disabled. Rename to a person works.

10. **Parenthetical strip-and-retry**
    - [ ] Type "Sean (development team)". Parenthetical stripped; resolver retries as "Sean".
    - [ ] If `Sean` exists as a contact: "Links to Sean". If near-match to "Sean Mooney": suggestion shown. If no match + eligible: "Will create new contact".

11. **Multi-person form rejected**
    - [ ] Type "Ed/Ellen". Hint: "Rename required: multi-person (contains /)".
    - [ ] User must rename to a single person (e.g., "Ed" or "Ellen").

12. **Contact mutation persists across daemon restart**
    - [ ] Complete a correction that creates a new contact.
    - [ ] Restart the daemon.
    - [ ] Verify the new contact survives (read `config.yaml`, check Settings UI).

13. **Live config refresh -- no restart required**
    - [ ] Start a correction flow in meeting A that creates a new contact.
    - [ ] Without restarting the daemon, start a new meeting B with the same participant name in enrichment.
    - [ ] Meeting B's enrichment `match_known_contacts` canonicalizes via the freshly-created contact (not via a restart-dependent cache).

14. **Legacy `.speakers.json` no-op + rewrite**
    - [ ] Place a pre-#28 `.speakers.json` file (keys = display labels like `Alice`, `Bob`) next to an older recording.
    - [ ] Reprocess the recording. Mapping silently no-ops (no crash, no error).
    - [ ] Open correction modal, save a new correction. `.speakers.json` now keyed by `speaker_id`.

15. **Clip playback after multiple corrections**
    - [ ] Correct `SPEAKER_00` to Alice. Play clip from Alice row. Works.
    - [ ] Re-correct Alice to Bob. Play clip from Bob row. Works (same audio, keyed by `speaker_id=SPEAKER_00`).
    - [ ] Correct Bob back to SPEAKER_00 (by clearing the mapping entry in `.speakers.json` or via modal). Play clip. Still works.
