Option A: New 'waiting' Field on Stage

Recommended

Add an optional waiting string field to PipelineStageStatus. Clean type-level separation from errors.

Python writes status.json
{
  "transcribe": { "completed": true, ... },
  "diarize":    { "completed": true, ... },
  "analyze":    {
    "completed": false,
    "waiting":   "speaker_review",
    "timestamp": null,
    "error":     null
  },
  "export":     { "completed": false, ... }
}
→
Rust reads & passes to frontend
// types.rs — add one field
pub struct PipelineStageStatus {
    pub completed: bool,
    pub timestamp: Option<String>,
    pub error: Option<String>,
    pub waiting: Option<String>,
}

// tauri.ts — add to interface
interface PipelineStageStatus {
  completed: boolean;
  timestamp: string | null;
  error: string | null;
  waiting: string | null;
}
→
Svelte renders dot state
// PipelineDots.svelte logic
function dotState(stage) {
  if (stage.completed) return 'done';
  if (stage.waiting)   return 'waiting';
  if (stage.error)     return 'error';
  return 'pending';
}

// Badge logic
const needsReview = Object.values(status)
  .some(s => s.waiting === 'speaker_review');
Strengths
  • Type-safe: waiting and error are distinct fields
  • Frontend check is simple: stage.waiting != null
  • Extensible: other wait reasons in the future
  • No string-parsing conventions to maintain
Trade-offs
  • Schema change in Python, Rust, and TypeScript
  • Existing status.json files lack the field (needs default)

Option B: Special Error String Convention

Use the existing error field with a WAITING: prefix. No schema changes needed.

Python writes status.json
{
  "transcribe": { "completed": true, ... },
  "diarize":    { "completed": true, ... },
  "analyze":    {
    "completed": false,
    "timestamp": null,
    "error":     "WAITING:speaker_review"
  },
  "export":     { "completed": false, ... }
}
→
Rust reads & passes to frontend
// No Rust type changes needed
pub struct PipelineStageStatus {
    pub completed: bool,
    pub timestamp: Option<String>,
    pub error: Option<String>,
}

// No TypeScript type changes needed
interface PipelineStageStatus {
  completed: boolean;
  timestamp: string | null;
  error: string | null;
}
→
Svelte renders dot state
// PipelineDots.svelte logic
function dotState(stage) {
  if (stage.completed) return 'done';
  if (stage.error) {
    if (stage.error.startsWith('WAITING:'))
      return 'waiting';
    return 'error';
  }
  return 'pending';
}

// Badge logic
const needsReview = Object.values(status)
  .some(s => s.error === 'WAITING:speaker_review');
Strengths
  • Zero schema changes — works with existing types
  • Backward compatible with old status.json files
  • Quick to implement
Trade-offs
  • Conflates waiting with errors — semantically wrong
  • String-prefix parsing is fragile convention
  • PipelineDots error display would show "WAITING:..." if logic missed
  • Retry button logic must also check for the prefix

Option C: Separate Top-Level 'paused_at' Key

Add a top-level paused_at object to status.json. Per-stage schema stays untouched.

Python writes status.json
{
  "paused_at": {
    "stage":  "analyze",
    "reason": "speaker_review"
  },
  "transcribe": { "completed": true, ... },
  "diarize":    { "completed": true, ... },
  "analyze":    {
    "completed": false,
    "timestamp": null,
    "error":     null
  },
  "export":     { "completed": false, ... }
}
→
Rust reads & passes to frontend
// types.rs — new struct + field
pub struct PausedAt {
    pub stage: String,
    pub reason: String,
}

pub struct PipelineStatus {
    pub paused_at: Option<PausedAt>,
    pub merge: PipelineStageStatus,
    // ...
}

// tauri.ts
interface PipelineStatus {
  paused_at: {stage: string, reason: string} | null;
  merge: PipelineStageStatus;
  // ...
}
→
Svelte renders dot state
// PipelineDots.svelte logic
function dotState(stageName, stage) {
  if (stage.completed) return 'done';
  if (status.paused_at?.stage === stageName)
    return 'waiting';
  if (stage.error) return 'error';
  return 'pending';
}

// Badge logic
const needsReview =
  status.paused_at?.reason === 'speaker_review';
Strengths
  • Per-stage schema completely unchanged
  • Clear separation: pause is a pipeline-level concept
  • Badge check is very clean
Trade-offs
  • Two places to check: per-stage fields + top-level paused_at
  • Must clear paused_at when pipeline resumes
  • PipelineStatus type changes in Rust + TypeScript
  • Stage doesn't know it's paused — indirection
Recommendation: Option A
The waiting field is the cleanest contract. It's one field added to one struct, propagated through three layers. The frontend logic reads naturally (if stage.waiting), and there's no string parsing or cross-referencing. The schema change is minimal — a single optional field that defaults to null for old status.json files.
